# bg-reviewer-pr-summary

Render a PR description for the change set produced this run. Write it to `artifacts/ghu/reviews/<run-timestamp>/pr-summary.md` so reviewers (or CI gh-cli automation) can paste it directly into the pull request.

## Sections (in order)

1. **Summary** — 1–3 bullet points of what changed and why, in user-facing language.
2. **Why** — link back to the approved spec section that motivated the change. Quote sparingly.
3. **Changes** — list of files modified or created, grouped by package. Skip generated files.
4. **Tests** — list of new test files / new test cases. Note any uncovered new code.
5. **Findings (severity ≥ medium)** — copy from `bg-reviewer-critique` output. Group by category (spec / constitution / idioms / test-quality / security / perf). Cite file + line range.
6. **Open Questions** — if any acceptance criterion is ambiguous or any agent stopped mid-task, list it here so the human reviewer can resolve.

## Style

- Tone: factual, neutral. No marketing language.
- No emoji. No "Generated by" attribution.
- One short paragraph per section, never more than the necessary.
